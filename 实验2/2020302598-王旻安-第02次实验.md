## 3. 文档编码规则

#### 3.1 目的

本文档是用于规范软件工程文档编码的指导性规范。本规范主要目的是保证软件工程文档的编写规范、规范化程度，提高软件工程文档编写的质量。确立文件分类与编码规则，便于文件追溯管理。

#### 3.2 范围

本规范适用于所有本项目所有文档的编写，包括所有内部与外部文档。

#### 3.3 通用要求

##### 3.3.1 文件命名规范

- 文件名应简明、具有描述性。
- 文件名应使用英文缩写与中文字词，用下划线（_）分隔。
- 文件名应包含文件类型后缀，如.doc，.pdf，.txt等，本项目默认使用.md。

##### 3.3.2 文件格式规范

- 文件应使用UTF-8编码。
- 文件应使用常规字体和字号，如宋体12号。
- 文件中的代码片段应使用等宽字体，如Consolas, Open Sans，以GitHub对markdown自动解析为准。

##### 3.3.3 文件结构规范

- 文件应包含标题、作者、日期等基本信息。
- ~~文件应包含目录，便于查找内容。请使用[TOC]引入(有待测试，typora使用段落-内容目录引入)~~已使用docsify自动生成目录

- 文件应包含章节、小节，便于阅读。章节和小节的标号请使用markdown默认的1-6级标题，同时添加数字。(按住ctrl+数字键引入)

##### 3.3.4 文档编写规范

- 文档应简洁明了、言简意赅，避免冗长、复杂。
- 文档应具有描述性、准确性，避免模糊、歧义。
- 文档应使用标准的技术术语，避免使用不规范的术语。
- 引入代码段请使用ctrl+shift+k，或\`\`\`引入。
- 表示引用请使用ctrl+shift+Q，使用引用块。
- 图片插入时请使用图床，绘图允许使用Mermaid，已配置有关解析
- 关于公式请使用latex标准语法，\$行内引入或\$\$块引入

##### 3.3.5 文档部署

+ 使用GitHub Pages+Docsify自动部署，已添加插件

#### 3.4 内部文档要求

##### 3.4.1 项目文档与开发文档

项目文档与开发文档命名应由3部分组成

包含**项目名称或缩写\_文档名称或缩写\_文档版本号**

文档版本号请使用带一位小数的版本

例如：新蜂商城项目的结构设计文档第一版命名可以为

新蜂商城\_架构设计\_V1.0

也可以为NEWBEEMALL_SDD_V1.0

为了见名知意，我不赞成用缩写表示文档名称。如需使用请全部大写。

本次默认**使用中文名称**

##### 3.4.2 过程文档

项目文档与开发文档命名也应由3部分组成

包含**项目名称或缩写\_文档名称\_日期(yyyy-MM-dd)**

例如：新蜂商城项目的2022.12.15的每日站立会议文档可行明文

新蜂商城\_每日站立会议\_2022-12-15

##### 3.4.3 内部文档样例说明

+ 大标题、编写人员与时间

  markdown支持直接使用html,css与js

  请使用以下代码直接引入文件标题

  ```html
  <div align="center">
      <h1>
          监所警员心理保障与管理系统_软件开发计划_V1.0
      </h1>
      <h3>
          编写人员:王旻安,党艳,吴思赣    编写时间:2023-3-25
      </h3>
  </div>
  ```

  为避免出现解析问题，编写人员与编写时间时间请使用**四个空格**而不是tab键。

  请在标题使用英文(半角)冒号与逗号，日期格式为yyyy-MM-dd

+ 文档评审

  请使用如下表格，与标题之间空一行

  

  **文档评审记录**

  | 日期 | 文档版本 | 评审说明 | 评审人 |
  | ---- | -------- | -------- | ------ |
  |      |          |          |        |
  |      |          |          |        |
  |      |          |          |        |

  

  表格结束后请空一行再开始正文内容

+ 段落与小标题

  请参考本文编写情况，依次使用2-5级标题。因解析原因请减少使用6级标题。

#### 3.5 外部文档要求

同内部文档中项目文档与开发文档

包含**项目名称或缩写\_文档名称或缩写\_文档版本号**三部分
